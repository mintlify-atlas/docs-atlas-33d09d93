By the end of this guide you will have Kami installed in your AI agent, a filled template ready to render, and a verified PDF on disk. The whole process takes under ten minutes for a first document, and subsequent documents require only steps 3 and 4.

<Note>
  Kami renders HTML templates to PDF using [WeasyPrint](https://weasyprint.org/). Install it along with `pypdf` before running the build script:

  ```bash
  pip install weasyprint pypdf --break-system-packages
  ```
</Note>

<Steps>
  <Step title="Install the skill">
    Add the Kami skill to your AI agent with a single command. Choose the variant that matches your agent:

    <CodeGroup>
    ```bash Claude Code
    npx skills add tw93/kami -a claude-code -g -y
    ```

    ```bash Codex
    npx skills add tw93/kami -a codex -g -y
    ```

    ```bash Generic agents
    npx skills add tw93/kami -a '*' -g -y
    ```
    </CodeGroup>

    The generic agents command (`-a '*'`) installs to `~/.agents/`, which is read by opencode, pi, and other tools that follow that convention.

    Once installed, the skill auto-triggers when you describe a document task ŌĆö no slash command required. You can also trigger it from Claude Desktop by downloading the ZIP from [Releases](https://github.com/tw93/kami/releases) and uploading it under Customize > Skills > "+".
  </Step>

  <Step title="Fill the template">
    Copy the matching template into your working directory, then edit only the body content. The CSS must stay untouched.

    ```bash
    cp assets/templates/one-pager-en.html ./my-document.html
    ```

    Open the file and replace all `{{placeholder}}` values with your content. The `--verify` command will flag any remaining placeholders before building.

    Key rules while editing:
    - **Edit body content only** ŌĆö replace text, metrics, section copy, and dates.
    - **Never touch the `<style>` block** ŌĆö all design decisions are already encoded there.
    - Follow `references/writing.en.md` for section structure: data over adjectives, distinctive phrasing over industry clich├®s.

    <Info>
      If you are handing raw material (meeting notes, a brain dump, or a chat transcript) to the agent, it will extract facts, map them to template sections, identify any gaps, and ask you once to fill them before proceeding.
    </Info>
  </Step>

  <Step title="Build and verify">
    Run `build.py` with the `--verify` flag to build the PDF and run all checks in a single pass:

    ```bash
    python3 scripts/build.py --verify
    ```

    To verify a single target instead of all templates:

    ```bash
    python3 scripts/build.py --verify resume-en
    ```

    The `--verify` command runs these checks in sequence:

    1. **Source file exists** ŌĆö confirms the template file is present.
    2. **Placeholder scan** ŌĆö reports any `{{...}}` values that were not filled.
    3. **WeasyPrint render** ŌĆö builds the PDF.
    4. **Page count check** ŌĆö warns if the output exceeds the document's page limit (one-pager: 1, letter: 1, resume: 2).
    5. **Font embedding check** ŌĆö confirms that the primary font is embedded. Chinese documents expect TsangerJinKai02; English documents expect Newsreader and Inter. Missing fonts trigger a fallback warning.

    A clean run looks like:

    ```
    Ō£ō one-pager-en: ok
    Ō£ō resume-en: ok
    ```

    A failing run names the problem directly:

    ```
    Ō£Ś resume-en: page overflow: 3 pages (limit 2)
    Ō£Ś one-pager-en: unfilled placeholder(s): {{company_name}}, {{tagline}}
    ```

    <Tip>
      Use `--check` (without `--verify`) to scan for CSS rule violations without building any PDFs. This is fast and useful during active editing:

      ```bash
      python3 scripts/build.py --check
      ```

      It reports `rgba()` backgrounds, cool-gray hex values, and line-heights above 1.55 ŌĆö the three most common invariant violations.
    </Tip>
  </Step>
</Steps>
